# $deep-interview 深度采访指南

> 苏格拉底式需求澄清专家

## 概述

`$deep-interview` 是 OMQ 中的**需求澄清**工作流。它通过苏格拉底式提问，将模糊想法转化为可执行的规格说明。

核心价值：
- **量化模糊度**：每轮计算模糊度分数（0-1）
- **压力测试**：每个答案都要证据/例子/假设检验
- **渐进澄清**：每轮只问一个最关键问题
- **可执行输出**：产出 `.omq/specs/deep-interview-*.md`

## 核心能力

### 1. 深度配置

| 配置 | 触发 | 目标阈值 | 最大轮数 | 用途 |
|------|------|---------|---------|------|
| **Quick** | `--quick` | ≤ 0.30 | 5 | 快速预 PRD 采访 |
| **Standard** | `--standard`（默认） | ≤ 0.20 | 12 | 标准需求采访 |
| **Deep** | `--deep` | ≤ 0.15 | 20 | 高严谨探索 |
| **Autoresearch** | `--autoresearch` | ≤ 0.20 | 12 | 研究任务准备 |

### 2. 澄清维度

**Greenfield（绿场）**：
```
模糊度 = 1 - (
  意图 × 0.30 +
  结果 × 0.25 +
  范围 × 0.20 +
  约束 × 0.15 +
  成功 × 0.10
)
```

**Brownfield（棕场）**：
```
模糊度 = 1 - (
  意图 × 0.25 +
  结果 × 0.20 +
  范围 × 0.20 +
  约束 × 0.15 +
  成功 × 0.10 +
  上下文 × 0.10
)
```

**维度说明**：
| 维度 | 权重 | 问题示例 |
|------|------|---------|
| 意图清晰度 | 30% | 为什么想要这个？ |
| 结果清晰度 | 25% | 最终状态是什么？ |
| 范围清晰度 | 20% | 变更走多远？ |
| 约束清晰度 | 15% | 技术/业务限制？ |
| 成功标准 | 10% | 如何评判完成？ |
| 上下文清晰度 | 10% | 现有代码理解（棕场） |

### 3. 阶段优先级

**Stage 1 — 意图优先**：
```
1. 意图
2. 结果
3. 范围
4. 非目标
5. 决策边界
```

**Stage 2 — 可行性**：
```
6. 约束
7. 成功标准
```

**Stage 3 — 棕场落地**：
```
8. 上下文清晰度
```

### 4. 挑战模式

| 模式 | 触发条件 | 用途 |
|------|---------|------|
| **Contrarian** | Round 2+ 或未测试假设 | 挑战核心假设 |
| **Simplifier** | Round 4+ 或范围扩张快于结果 | 探索最小可行范围 |
| **Ontologist** | Round 5+ 且模糊度>0.25 | 本质级重构 |

### 5. 执行桥

采访完成后提供执行选项：

| 选项 | 调用 | 用途 |
|------|------|------|
| **$ralplan** | 推荐 | 共识规划 |
| **$autopilot** | 直接执行 | 端到端自主 |
| **$ralph** | 持久执行 | 循环直到完成 |
| **$team** | 协调并行 | 团队协作 |
| **继续细化** | 继续采访 | 进一步澄清 |

## 工作流程

### Phase 0: 上下文摄入

**目标**：建立 grounded context

**流程**：
```
1. 派生任务 slug
2. 加载/创建上下文快照
   - 路径：`.omq/context/{slug}-{timestamp}.md`
   - 内容：任务声明、期望结果、已知事实、约束、未知问题、可能触点

3. 记录路径到模式状态
```

### Phase 1: 初始化

**目标**：设置采访参数

**流程**：
```
1. 解析参数和深度配置
2. 检测项目类型（绿场/棕场）
   - 绿场：无现有代码
   - 棕场：运行 explore 收集上下文

3. 初始化状态
   ```json
   {
     "active": true,
     "profile": "standard",
     "type": "brownfield",
     "current_ambiguity": 1.0,
     "threshold": 0.20,
     "max_rounds": 12,
     "current_stage": "intent-first",
     "current_focus": "intent"
   }
   ```

4. 宣布开始（配置、阈值、当前模糊度）
```

### Phase 2: 采访循环

**目标**：逐轮降低模糊度

**每轮流程**：
```
1. 生成下一个问题
   - 基于：原始想法、 prior Q&A、当前维度分数、棕场上下文
   - 目标：最弱维度（尊重阶段优先级）

2. 提问（使用结构化输入工具）
   ```
   Round {n} | 目标：{weakest_dimension} | 模糊度：{score}%
   
   {问题}
   ```

3. 评分模糊度
   - 每个维度 [0.0, 1.0]
   - 提供理由和差距

4. 报告进度
   - 显示加权分解表
   - 准备门状态（非目标、决策边界）
   - 下一焦点维度

5. 持久化状态
   - 追加轮次结果和更新分数

6. 压力测试答案
   - 要求证据/例子
   - 暴露假设
   - 强制边界/权衡
   - 症状 vs 根因重构
```

**压力测试梯子**：
```
1. 要求证据/例子/反例
2. 探索隐藏假设/依赖/信念
3. 强制边界/权衡：明确不做/推迟/拒绝什么
4. 症状 → 根因重构
```

### Phase 3: 挑战模式

**目标**：压力测试假设

**模式触发**：
```
Contrarian:
  - Round 2+ 或
  - 答案基于未测试假设

Simplifier:
  - Round 4+ 或
  - 范围扩张快于结果清晰度

Ontologist:
  - Round 5+ 且模糊度>0.25 或
  - 用户持续描述症状
```

**每种模式使用一次**。

### Phase 4: 结晶

**目标**：产出采访 artifacts

**流程**：
```
1. 检查准备门
   - 非目标：必须明确
   - 决策边界：必须明确
   - 压力测试：至少完成一次

2. 写入采访记录
   - 路径：`.omq/interviews/{slug}-{timestamp}.md`

3. 写入可执行规格
   - 路径：`.omq/specs/deep-interview-{slug}.md`

4. 规格内容：
   - 元数据（配置、轮数、最终模糊度、阈值、上下文类型）
   - 上下文快照引用
   - 清晰度分解表
   - 意图（为什么）
   - 期望结果
   - 范围内
   - 范围外/非目标
   - 决策边界（OMQ 可自行决定什么）
   - 约束
   - 可测试验收标准
   - 暴露假设 + 解决方案
   - 压力测试发现
   - 棕场证据 vs 推断
   - 技术上下文发现
   - 完整/精简记录
```

### Phase 5: 执行桥

**目标**：提供执行选项

**选项**：

**1. $ralplan（推荐）**：

  - 不重复采访
  - 围绕澄清的意图细化架构/可行性
输出：.omq/plans/prd-*.md, .omq/plans/test-spec-*.md
最佳：需求清晰但需要架构验证
```

**2. $autopilot**：
```
输入：.omq/specs/deep-interview-{slug}.md
调用：$autopilot <spec-path>
消费行为：
  - 使用规格作为澄清的执行简报
  - 保留意图、非目标、决策边界、验收标准
最佳：规格足够强可直接规划 + 执行
```

**3. $ralph**：
```
输入：.omq/specs/deep-interview-{slug}.md
调用：$ralph <spec-path>
消费行为：
  - 使用规格的验收标准和边界约束作为持久目标
  - 不重新打开需求发现
最佳：需要持久顺序完成压力
```

**4. $team**：
```
输入：.omq/specs/deep-interview-{slug}.md
调用：$team <spec-path>
消费行为：
  - 视为共享执行上下文
  - 保留澄清的意图、非目标、决策边界、验收标准
最佳：大型、多通道、阻塞敏感任务
```

**5. 继续细化**：
```
输入：现有记录、上下文快照、当前规格草稿
调用：继续采访循环
输出：更低模糊度规格
最佳：剩余模糊度仍太高
```

## 最佳实践

### 1. 每轮只回答一个问题

❌ **批量回答**：
```
用户：主要目标是 X，目标用户是 Y，技术栈是 Z...
```

✅ **逐轮回答**：
```
Planner: 主要目标是什么？
用户：X

Planner: 目标用户是谁？
用户：Y
```

### 2. 棕场先探索再提问

❌ **问已知**：
```
Planner: 认证在哪里实现？
```

✅ **先探索**：
```
Planner: [explore 查找认证实现]
Planner: 我看到认证在 src/auth/。应该扩展现有逻辑还是新建模块？
```

### 3. 压力测试每个答案

❌ **表面接受**：
```
用户：需要高性能
Planner: 好的（继续）
```

✅ **压力测试**：
```
用户：需要高性能
Planner: 具体指标是什么？p50 还是 p99？目标值？
```

### 4. 非目标和决策边界必须明确

```
不要跳过非目标
不要跳过决策边界
即使模糊度达标也要明确
```

### 5. 使用执行桥

```
采访完成后选择执行路径
不要直接实现（deep-interview 是需求模式）
```

## 常见问题

### Q: 采访会问多少轮？

**A**: 取决于配置：
- Quick: 最多 5 轮
- Standard: 最多 12 轮
- Deep: 最多 20 轮

实际轮数取决于模糊度下降速度。

### Q: 可以跳过某些问题吗？

**A**: 可以说"跳过这个"，但会延长采访轮数。

### Q: 采访中途可以停止吗？

**A**: 可以。说"停止"或运行 `/cancel`。进度保存，可恢复。

### Q: 规格保存在哪里？

**A**: 
- 采访记录：`.omq/interviews/{slug}-{timestamp}.md`
- 可执行规格：`.omq/specs/deep-interview-{slug}.md`

### Q: 如何选择执行选项？

**A**: 
- 需要架构验证 → $ralplan
- 规格清晰直接执行 → $autopilot
- 需要持久压力 → $ralph
- 大型协作 → $team
