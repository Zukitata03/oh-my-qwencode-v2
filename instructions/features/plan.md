# $plan 工作流完整指南

> 战略规划与需求收集专家

<!-- omq:generated:skill-doc -->

## 概述

`$plan` 是 OMQ 中的**战略规划**工作流，通过智能检测自动选择最适合的规划模式：

| 模式 | 触发条件 | 行为 |
|------|---------|------|
| **采访模式** | 模糊/宽泛请求 | 交互式需求收集 |
| **直接模式** | `--direct` 或详细请求 | 直接生成计划 |
| **共识模式** | `--consensus` 或高风险 | 多角色审议直到一致 |
| **审查模式** | `--review` | 评估现有计划 |

---

## 核心能力

### 1. 智能模式检测

自动识别请求类型并选择最佳模式：

```
模糊请求 → 采访模式
"我想做个任务管理应用"

详细请求 → 直接模式
"用 TypeScript 重构 auth 模块，参考 RFC-123"

高风险 → 共识模式
"重构核心支付系统"
```

### 2. 交互式需求收集

采访模式下，逐轮收集需求：

```
Round 1: 主要目标是什么？
Round 2: 目标用户是谁？
Round 3: 技术栈偏好？
Round 4: 时间约束？
...
```

### 3. 多角色审议（共识模式）

Planner → Architect → Critic 循环审议：

```
1. Planner 创建初始计划
2. Architect 审查架构合理性
3. Critic 评估质量
4. 循环直到批准（最多 5 次）
```

### 4. 计划质量保障

所有计划必须满足：
- 80%+ 声明引用文件/行号
- 90%+ 验收标准可测试
- 所有风险有缓解措施
- 所有文件引用存在

## 使用方法

### 基本语法

```
$plan "<任务描述>"
$plan --direct "<详细任务>"
$plan --consensus "<高风险任务>"
$plan --review "<计划文件>"
```

### 模式组合

```
# 共识 + 直接（跳过采访）
$plan --consensus --direct "迁移数据库"

# 共识 + 交互（用户审批后执行）
$plan --consensus --interactive "重构认证系统"
```

### RALPLAN-DR 模式

共识模式支持两种审议强度：

| 模式 | 触发 | 用途 |
|------|------|------|
| **Short** | 默认 | 标准风险任务 |
| **Deliberate** | `--deliberate` | 高风险任务 |

**Deliberate 模式额外包含**：
- 预演失败（3 个失败场景）
- 扩展测试计划（单元/集成/e2e/可观测性）

## 工作流程

### 采访模式流程

```
1. 分类请求 → 模糊/宽泛触发采访
2. 探索代码库 → 先查找事实再提问
3. 问问题 → 每轮一个焦点问题
4. 咨询 Analyst → 发现隐藏需求
5. 创建计划 → 用户表示准备好
```

**示例对话**：

```
用户：$plan "我想做个任务管理应用"

Planner: [探索代码库，发现现有用户系统]
Planner: 我看到你已有用户系统。这个任务管理应用
         是面向内部团队还是外部客户？

用户：外部客户

Planner: 明白了。需要支持多租户吗？
         （每个客户独立的数据空间）

用户：需要

Planner: [继续收集需求...]
```

### 直接模式流程

```
1. 快速分析 → 可选 Analyst 咨询
2. 创建计划 → 直接生成完整计划
3. 审查 → 可选 Critic 审查
```

### 共识模式流程

```
1. Planner 创建初始计划 + RALPLAN-DR 摘要
   - 原则（3-5 个）
   - 决策驱动因素（top 3）
   - 可行选项（≥2 个，含 pros/cons）

2. 用户反馈（--interactive 模式）
   - 继续审查 → 送 Architect 和 Critic
   - 请求变更 → 返回步骤 1
   - 跳过审查 → 直接最终审批

3. Architect 审查
   - 最强反论（antithesis）
   - 权衡张力（tradeoff）
   - 综合路径（synthesis）

4. Critic 评估
   - 原则 - 选项一致性
   - 替代方案探索公平性
   - 风险缓解清晰度
   - 验收标准可测试性

5. 审查循环（最多 5 次）
   - Critic 拒绝 → Planner 修订 → 返回步骤 3
   - Critic 批准 → 继续步骤 6

6. 应用改进
   - 合并所有接受的改进
   - 添加 ADR 章节
   - 添加执行人员配置指导

7. 最终审批（--interactive 模式）
   - 批准并执行 → $ralph
   - 批准并团队执行 → $team
   - 请求变更 → 返回步骤 1
   - 拒绝 → 丢弃计划
```

### 审查模式流程

```
1. 读取计划文件
2. Critic 评估质量
3. 返回裁决：APPROVED / REVISE / REJECT
```

### 案例 2：高风险共识规划

**指令**：
```
$plan --consensus --deliberate "重构支付核心系统"
```

**RALPLAN-DR 摘要**：
```markdown
## 原则
1. 零数据丢失
2. 零财务误差
3. 合规第一（PCI DSS）

## 决策驱动因素
1. 财务数据准确性
2. 监管合规
3. 业务连续性

## 可行选项

### 选项 A：绞杀者模式
逐步替换旧系统，新旧并存一段时间。

**优点**：
- 风险可控
- 可随时回滚
- 业务无感知

**缺点**：
- 实现复杂
- 需要双写逻辑
- 临时技术债务

### 选项 B：一次性迁移

**优点**：
- 实现简单
- 无临时债务
- 测试集中

**缺点**：
- 需要停机
- 回滚风险高
- 业务受影响

## 无效化理由
选项 B 因停机风险和回滚风险过高，不符合原则 1 和 3，故排除。
```

```markdown
**预防**：提前审计 + 合规审查
```

## 常见问题

### Q: 采访模式会问多少轮问题？

**A**: 取决于需求清晰度，通常 5-10 轮。可随时说"我准备好了"结束采访。

### Q: 共识模式需要多长时间？

**A**: 通常 3-5 轮审议，每轮包含 Planner/Architect/Critic。复杂任务可能需要更久。

### Q: 可以中途切换模式吗？

**A**: 可以。采访模式下可以说"直接生成计划吧"切换到直接模式。

### Q: 计划保存在哪里？

**A**: `.omq/plans/` 目录下，文件名包含时间戳和任务关键词。

### Q: 如何恢复中断的规划？

**A**: 重新运行 `$plan`，会从 `.omq/state/` 恢复状态。
